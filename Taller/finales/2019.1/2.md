2)¿Qué significa que una función es blocante?¿Cómo subsanaría esa limitación en término de mantener el programa ‘vivo’ ?

Una funcion es bloqueante cuando el proceso que esta realizando no permite continuar con las demas tareas y las deja esperando a que termine. Una forma de subsanar esa limitacion sería correr esa funcion en un thread separado de forma paralela para que el resto del programa pueda continuar su proceso.

3)Explique qué son los métodos virtuales y para qué sirven. De un breve ejemplo donde su uso sea imprescindible

Un método virtual es un método declarado en una clase base con la palabra clave `virtual` que puede ser redefinido (override) por las clases derivadas. Su particularidad es que la decisión de qué implementación ejecutar no se toma en tiempo de compilación según el tipo estático del puntero/referencia, sino en tiempo de ejecución según el tipo dinámico (real) del objeto al que se apunta. Esto se conoce como polimorfismo dinámico o *late binding*, y se implementa internamente mediante una tabla de punteros a funciones (vtable).

Sirven para que el código que trabaja con punteros o referencias a la clase base pueda invocar el comportamiento específico de cada clase derivada sin necesidad de conocer su tipo concreto, permitiendo escribir código genérico y extensible (agregar nuevas clases derivadas no requiere modificar el código que las usa).

Ejemplo donde su uso es imprescindible: manejar una colección heterogénea de objetos a través de punteros a la clase base y que cada uno se comporte según su tipo real.

```C++
class Figura{
    public:
        virtual double area() const = 0; // método virtual puro
        virtual ~Figura() = default;      // destructor virtual, también imprescindible
};

class Circulo : public Figura{
    private:
        double radio;
    public:
        Circulo(double r) : radio(r) {}
        double area() const override { return 3.14159 * radio * radio; }
};

class Rectangulo : public Figura{
    private:
        double base, altura;
    public:
        Rectangulo(double b, double h) : base(b), altura(h) {}
        double area() const override { return base * altura; }
};

void imprimirArea(const Figura& f){
    // sin virtual, siempre llamaría a Figura::area (que ni siquiera tiene implementación)
    std::cout << f.area() << std::endl;
}

int main(){
    Circulo c(2);
    Rectangulo r(3, 4);
    imprimirArea(c); // llama a Circulo::area
    imprimirArea(r); // llama a Rectangulo::area
}
```

Si `area()` no fuera virtual, `imprimirArea` siempre invocaría la versión de `Figura` (binding estático), sin importar el tipo real del objeto pasado por referencia, perdiendo el comportamiento polimórfico.

4)Escribir un programa C que procese el archivo “numeros.txt” sobre sí mismo (sin crear archivos intermedios y sin subir el archivo a memoria). El procesamiento consiste en leer grupos de 4 caracteres hexadecimales y reemplazarlos por los correspondientes dígitos decimales (que representen el mismo número leído pero en decimal)

El truco del ejercicio es que 4 dígitos hexadecimales (0x0000 a 0xFFFF) pueden representar un número de 1 a 5 dígitos decimales, así que el reemplazo casi nunca mide 4 caracteres. Por eso, cada vez que cambia el largo hay que correr el resto del archivo esa diferencia (`delta`) de posiciones antes de escribir. Como no puedo cargar todo a memoria, lo hago byte a byte con `fseek`/`fgetc`/`fputc`.

```C
#include <stdio.h>
#include <unistd.h>

int hexval(char c){
    if (c >= '0' && c <= '9') return c - '0';
    return (c | 32) - 'a' + 10; // pasa a minuscula y calcula
}

// corre el contenido desde 'desde' hasta el final del archivo, delta posiciones
void correr(FILE *f, long desde, long tam, int delta){
    if (delta > 0){
        for (long i = tam - 1; i >= desde; i--){
            fseek(f, i, SEEK_SET);
            int c = fgetc(f);
            fseek(f, i + delta, SEEK_SET);
            fputc(c, f);
        }
    } else if (delta < 0){
        for (long i = desde; i < tam; i++){
            fseek(f, i, SEEK_SET);
            int c = fgetc(f);
            fseek(f, i + delta, SEEK_SET);
            fputc(c, f);
        }
        ftruncate(fileno(f), tam + delta);
    }
}

int main(int argc, char *argv[]){
    FILE *f = fopen(argv[1], "r+b");
    if (f == NULL) return 1;

    fseek(f, 0, SEEK_END);
    long tam = ftell(f);

    long pos = 0;
    while (pos + 4 <= tam){
        char hex[4];
        fseek(f, pos, SEEK_SET);
        fread(hex, 1, 4, f);
        int valor = hexval(hex[0]) * 4096 + hexval(hex[1]) * 256 + hexval(hex[2]) * 16 + hexval(hex[3]);

        char dec[6];
        int len = sprintf(dec, "%d", valor);

        if (len != 4){
            correr(f, pos + 4, tam, len - 4);
            tam += len - 4;
        }

        fseek(f, pos, SEEK_SET);
        fwrite(dec, 1, len, f);
        pos += len;
    }

    fclose(f);
    return 0;
}
```

5)Explique qué se entiende por “compilación condicional”. Ejemplifique mediante código.

Es incluir o excluir partes del código fuente antes de compilar, usando directivas del preprocesador (`#ifdef`, `#ifndef`, `#if`, `#else`, `#endif`) según constantes definidas (por ejemplo con `#define` o `-D` al compilar). Sirve para tener versiones distintas de un mismo código (debug/release, por sistema operativo, etc.) sin duplicar archivos.

```C
#include <stdio.h>

#define DEBUG 1

int main(){
#if DEBUG
    printf("modo debug\n");
#else
    printf("modo release\n");
#endif
    return 0;
}
```

6)Escriba un programa en C++ que tome 2 cadenas por línea de comandos: A y B; e imprima la cadena A después de haber duplicado todas las ocurrencias de B..
ej.: reemp.exe “Este es el final” final -----> Este es el final final

```C++
#include <iostream>
#include <string>

int main(int argc, char* argv[]){
    std::string A = argv[1];
    std::string B = argv[2];

    size_t pos = 0;
    while ((pos = A.find(B, pos)) != std::string::npos){
        A.insert(pos + B.size(), " " + B);
        pos += 2 * B.size() + 1; // salto la ocurrencia original y la copia insertada
    }

    std::cout << A << std::endl;
    return 0;
}
```

8 ) ¿Qué valor arroja sizeof(int)? Justifique .

Sizeof(int) arroja 4 en la gran mayoría de los compiladores y arquitecturas actuales (modelo LP64, típico en Linux/Mac de 64 bits), ya que un int ocupa 4 bytes (32 bits). El estándar de C++ no fija ese valor exacto: sólo garantiza que sizeof(int) >= sizeof(short) y que un int tiene al menos 16 bits, por lo que el resultado puede variar según la plataforma y el compilador (por ejemplo, en sistemas embebidos de 16 bits puede dar 2).

9 ) Describa el concepto de loop de eventos (events loop) utilizado en programación orientada
a eventos y, en particular, en entornos de interfaz gráfica (GUIs).

En los entornos de interfaz gráfica se suele utilizar el concepto de loop de eventos, que consiste en un bucle infinito que permanece escuchando una cola de eventos (clicks, teclas, redibujados, etc.) generados por el sistema operativo o el usuario. Por cada evento que llega, el loop lo saca de la cola en orden de llegada y lo despacha al handler o callback correspondiente para que lo procese; cuando la cola está vacía, el loop queda bloqueado esperando el próximo evento en lugar de consumir CPU. Esto permite que una aplicación con un único hilo principal reaccione a múltiples fuentes de eventos sin necesidad de que el programador esté sondeando (polling) manualmente.

10 ) ¿Qué ventaja ofrece un lock raii frente al tradicional lock/unlock?

Un lock RAII (por ejemplo `std::lock_guard` o `std::unique_lock`) toma el lock en su constructor y lo libera automáticamente en su destructor, cuando el objeto sale de scope. Esto ofrece varias ventajas frente a llamar `lock()`/`unlock()` a mano: garantiza que el lock se libere incluso si en el medio se lanza una excepción o hay un `return` anticipado (evitando deadlocks por olvidos), y hace el código más simple de leer y de mantener, ya que no depende de que el programador recuerde liberar el lock en todos los caminos posibles de la función.

Un lock raii frente a un lock tradicional ofrece mayor seguridad a la hora de liberar ya que no se debe hacer a mano, mayor facilidad de lectura y mayor facilidad a la hora de rehacer o modificar codigo
